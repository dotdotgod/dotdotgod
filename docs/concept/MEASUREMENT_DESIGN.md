# Measurement Design
## Purpose
This document defines how to measure dotdotgod's context curation effects without overstating claims.

dotdotgod is not primarily a token optimizer. Measurements should separate token totals from context quality outcomes such as task continuity, fewer repeated explanations, lower scope drift, and verification success.

## Metrics
### Loader Prompt Size
Estimate the size of the prompt generated by `/dd:load`:

- characters
- words
- approximate tokens, using `characters / 4` as a rough local estimate unless a model-specific tokenizer is available

This measures fixed workflow overhead.

### Memory Surface Size
Estimate the size of files that the agent is asked to consider first:

- `AGENTS.md`
- root `README.md`
- `docs/README.md`
- docs area indexes
- selected `docs/spec`, `docs/arch`, and `docs/test` files relevant to the task

This measures the curated project-memory surface.

### Archive Excluded Size
Estimate tokens avoided by excluding `docs/archive` from automatic directory summaries:

- token estimate for `docs/archive/README.md`
- token estimate for all archived plan/report markdown files
- token estimate for only targeted archives selected for a task

This shows how historical memory can remain available without dominating default context.

### Plan Compaction Effect
Record context usage around Plan Mode compaction:

- before Plan Mode compaction
- immediately after compaction when available
- after the next assistant response, because usage can be unavailable or incomplete immediately after compaction
- before execution begins

This measures whether compaction shaped a long planning session into a smaller continuation context.

### Useful-Context Density
For a representative task, manually classify loaded context chunks:

- directly relevant
- indirectly relevant
- irrelevant/noisy

Useful-context density can be reported as:

```text
(directly relevant + indirectly relevant tokens) / total loaded tokens
```

This is subjective, so keep the rubric and examples with the measurement.

### Repeated Explanation Count
Count how often the user must restate:

- project rules
- documentation naming conventions
- verification commands
- previous decisions
- active task intent
- archive policy

This measures whether curated memory reduces repeated user effort.

### Task Outcome Quality
Record task outcomes that matter more than raw token totals:

- plan completeness
- scope drift
- number of user corrections
- verification success
- rework count
- whether completed work was archived with useful future context

## Static Token Estimate Method
A simple local script can estimate token surfaces before introducing model-specific tokenizers.

Suggested inputs:

1. baseline memory files
2. docs indexes
3. selected spec/arch/test files
4. active plan files
5. archive files, separately measured as optional context

Suggested output:

| Group | Files | Characters | Approx tokens | Notes |
| --- | ---: | ---: | ---: | --- |
| Baseline | n | n | n | `AGENTS.md`, root README, docs indexes |
| Selected docs | n | n | n | Task-relevant spec/arch/test |
| Active plans | n | n | n | Current task intent |
| Archive index | n | n | n | Available history map |
| Full archive | n | n | n | Not loaded by default |

Use approximate tokens only as directional evidence. Label them as estimates.

## Pi Runtime Observation Method
Use Pi's context usage reporting when available.

Observation points:

1. before `/dd:load`
2. after `/dd:load`
3. before enabling Plan Mode
4. after Plan Mode writes or updates a plan
5. immediately before Plan Mode compaction
6. after compaction and the next assistant response

## Scenario Benchmark Method
Run repeated scenarios with and without the dotdotgod workflow.

Suggested scenarios:

1. small documentation change
2. feature change touching source and docs
3. architecture-sensitive change requiring conventions
4. long planning session with multiple alternatives
5. follow-up task that should reuse an archived decision

For each scenario, record:

| Scenario | Workflow | Load tokens | Peak context | Compactions | User restatements | Verification | Notes |
| --- | --- | ---: | ---: | ---: | ---: | --- | --- |

Workflow examples:

- ad-hoc chat only
- dotdotgod load + docs-first planning
- dotdotgod load + Plan Mode + archive lookup

Keep model, date, package version, and prompt wording fixed where possible.

## Archive Growth Experiment
Historical memory is useful but can become noisy if loaded indiscriminately.

Measure the same repository at increasing archive sizes:

- no archive
- archive index only
- 10 archived plans
- 50 archived plans
- all archived plans
- targeted archive selected by task

Compare:

- directory summary size
- prompt size
- approximate loaded tokens
- time spent locating relevant history
- whether the agent selected the correct archive

This validates the current design choice: archives are available through indexes and targeted lookup, but excluded from the default documentation directory summary.

## Reporting Guidance
Do not report a single number as proof of token savings. Prefer paired claims:

- token/context usage changed by X in this scenario
- useful-context density changed by Y under this rubric
- user restatements changed from A to B
- verification/rework outcome changed from C to D

Example wording:

```text
In this scenario, dotdotgod added a small loader overhead but avoided loading the full archive by default. The agent still found the relevant plan through the archive index, and the user did not need to restate the project naming rules.
```

## Guardrails
- Do not compare different models without labeling model and context window.
- Do not claim universal token savings from one project or one task.
- Separate fixed workflow overhead from long-session savings.
- Separate total tokens from useful-context density.
- Treat `characters / 4` as an estimate, not an exact tokenizer result.
- Keep measurement scripts optional until they are promoted into package tooling.
